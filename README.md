# 翻译助手工具介绍使用说明
## 1. 工具简介

`翻译助手 MVP` 是一个基于 **Chrome Manifest V3** 开发的浏览器翻译插件，当前版本已经具备以下核心能力：

- `Popup 文本翻译`
- `网页划词翻译`
- `整页正文翻译`
- `恢复网页原文`
- `快捷键触发翻译`
- `本地保存 API 配置`

该工具通过 **OpenAI 兼容的 Chat Completions 接口** 完成翻译请求，因此不仅可以接入 OpenAI，也可以接入兼容该接口格式的其他模型服务。

### 2.1 目录职责说明

#### `manifest.json`

插件主配置文件，定义了：

- 插件名称、版本、描述
- popup 入口：`popup/popup.html`
- 后台脚本：`background/service-worker.js`
- 内容脚本：`content/content.js`
- 样式注入：`content/content.css`
- 权限配置：`storage`、`tabs`、`scripting`
- 快捷键：`Ctrl+Shift+Y` / `Command+Shift+Y`
- 网页访问范围：`http/https/localhost`

#### `background/`

后台逻辑目录，核心文件是 `service-worker.js`。

主要职责：

- 接收 popup 和 content script 发来的消息
- 统一调用翻译接口
- 处理整页翻译和恢复请求
- 监听快捷键命令
- 按需向当前标签页注入 content script

它相当于整个插件的“调度中心”。

#### `content/`

网页内容注入脚本目录。

`content.js` 主要负责：

- 监听网页选区变化
- 读取用户当前选中的文本
- 请求后台执行翻译
- 在页面上渲染浮层显示译文
- 扫描正文文本并执行整页翻译
- 在需要时恢复原始网页文本

`content.css` 负责翻译浮层的样式，包括：

- 浮层卡片
- 状态颜色
- 标题与正文排版
- 错误态与加载态表现

`下一步-划词翻译.md` 是开发过程中的阶段性设计记录，说明该模块最初的拆分目标。

#### `popup/`

插件弹窗界面目录。

`popup.html` 提供用户操作界面，包含：

- API Key 输入框
- API Base URL 输入框
- 模型名称输入框
- 目标语言选择框
- 文本翻译区
- 整页翻译按钮
- 恢复原文按钮

`popup.js` 负责：

- 初始化读取已保存配置
- 保存用户设置
- 发送文本翻译请求
- 发送整页翻译请求
- 发送恢复原文请求
- 在界面中展示成功或失败状态

`popup.css` 负责整个弹窗的视觉样式。

#### `providers/`

接口适配层目录。

当前只有一个实现文件：`openai-compatible.js`，说明当前工具采用的是“OpenAI 兼容接口”方案。

主要职责：

- 规范化 Base URL
- 构造 `/chat/completions` 请求
- 注入系统提示词，要求模型只返回译文
- 处理超时和接口报错
- 提取模型返回内容
- 提供单段翻译与批量翻译能力

#### `shared/`

公共模块目录。

`messages.js`：

- 统一管理消息类型常量
- 避免 popup、background、content 之间消息名称不一致

`storage.js`：

- 定义默认配置
- 读取本地配置
- 保存本地配置

这部分是项目的“共享基础设施”。

#### 根目录中的测试与文档文件

- `popup-e2e-result.json`：popup 翻译链路测试结果
- `selection-e2e-result.json`：划词翻译测试结果
- `page-e2e-result.json`：整页翻译与恢复测试结果
- `会议纪要-Chrome翻译插件开发规划.md`：项目规划文档

这些文件说明该项目不仅有实现代码，也保留了开发规划和功能验证结果。

## 3. 工具工作原理

### 3.1 Popup 翻译链路

用户在插件弹窗中输入文本后，流程如下：

1. 在 popup 中填写配置和待翻译文本
2. `popup.js` 保存当前设置
3. `popup.js` 向 `service-worker.js` 发送翻译消息
4. `service-worker.js` 读取配置并调用 `providers/openai-compatible.js`
5. 翻译结果返回 popup
6. popup 显示译文和状态提示

### 3.2 划词翻译链路

用户在网页中选中文本后，流程如下：

1. `content.js` 监听到选区变化或鼠标抬起
2. 读取选中文本
3. 向后台发送 `translate-text` 请求
4. 后台调用模型接口翻译
5. content script 在页面中显示翻译浮层

### 3.3 整页翻译链路

用户点击 popup 中的 `整页翻译` 后，流程如下：

1. popup 向后台发送整页翻译请求
2. 后台将请求转发到当前网页标签页
3. `content.js` 扫描页面正文文本节点
4. 按顺序批量请求翻译
5. 翻译完成后直接替换网页中的原文本
6. popup 或页面浮层提示翻译完成

### 3.4 恢复原文链路

整页翻译后，`content.js` 会缓存原始文本节点内容。点击 `恢复原文` 时会：

1. 向当前页面发送恢复请求
2. 将已替换的节点文本恢复为原值
3. 清空缓存状态

## 4. 当前版本已实现功能

### 4.1 基础设置

支持以下配置项：

- `API Key`
- `API Base URL`
- `Model`
- `目标语言`

默认值如下：

- `Base URL`：`https://api.openai.com/v1`
- `Model`：`gpt-4o-mini`
- `目标语言`：`简体中文`

### 4.2 Popup 文本翻译

用户可在插件弹窗中输入任意文本并直接翻译，适合：

- 句子翻译
- 段落翻译
- 临时复制文本翻译

### 4.3 网页划词翻译

在普通网页中选中文本后，会自动或快捷键触发翻译，并以浮层形式展示结果。

特点：

- 不跳转页面
- 不打断阅读
- 支持 `Esc` 关闭浮层

### 4.4 快捷键翻译

支持快捷键：

- Windows：`Ctrl + Shift + Y`
- macOS：`Command + Shift + Y`

适合在鼠标选中文本后快速触发翻译。

### 4.5 整页翻译

支持对页面中筛选出的正文文本节点执行整页翻译，适合快速阅读外文页面。

当前实现特征：

- 只处理符合条件的文本节点
- 会跳过脚本、样式、输入框、代码块等区域
- 对过短、过长或纯数字类文本会自动忽略

### 4.6 恢复原文

执行整页翻译后，可以一键恢复网页原始文本。

## 6. 使用方法

### 6.1 首次使用配置

首次打开插件后，请先在 popup 中填写：

- `API Key`
- `API Base URL`
- `Model`
- `目标语言`

然后点击 `保存设置`。

如果你使用的是 OpenAI 官方接口，可参考：

- `API Base URL`：`https://api.openai.com/v1`
- `Model`：`gpt-4o-mini`

如果你使用的是其他 OpenAI 兼容服务，则将 `Base URL` 和 `Model` 替换为对应平台提供的值即可。

### 6.2 使用 Popup 文本翻译

1. 点击浏览器中的插件图标
2. 在 `待翻译文本` 输入框中输入内容
3. 点击 `翻译文本`
4. 在 `译文` 区域查看结果

### 6.3 使用划词翻译

1. 打开任意普通网页
2. 用鼠标选中一段文本
3. 等待自动翻译浮层出现

如果没有自动触发，也可以按快捷键：

- `Ctrl + Shift + Y`

### 6.4 使用整页翻译

1. 打开要阅读的网页
2. 点击插件图标
3. 点击 `整页翻译`
4. 返回网页等待内容更新

### 6.5 恢复网页原文

1. 点击插件图标
2. 点击 `恢复原文`
3. 当前页会恢复为翻译前文本

## 9. 已知实现特点与限制

### 9.1 接口依赖外部模型服务

该工具本身不提供翻译引擎，必须依赖外部兼容接口，因此：

- 没有 API Key 时无法使用
- 接口超时或服务异常时会翻译失败

### 9.2 整页翻译为轻量实现

当前整页翻译并不是完整 DOM 级别的专业网页翻译引擎，而是：

- 选择部分正文文本节点进行替换
- 控制节点数量和文本长度，避免一次请求过重

因此在复杂网站中，可能出现：

- 部分区域未翻译
- 特殊布局页面效果不完全一致

### 9.3 批量翻译当前采用串行方式

`translateTextBatch()` 当前是逐条调用接口，而不是一次性真正并发批量提交，因此在整页翻译场景下：

- 页面越长，耗时可能越明显
- 接口响应速度会直接影响使用体验

### 9.4 仅适用于普通网页场景

插件当前主要面向：

- `http://` 页面
- `https://` 页面

某些浏览器内部页面、扩展页、受限页面可能无法正常注入脚本。
